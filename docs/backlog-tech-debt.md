**現狀**

目前 business data（`lot_status`、`sales_order`）和 authz policy store（所有 `authz_*` 表）放在同一個 PostgreSQL DB（`nexus_authz`）。這是 POC 階段的設計決策，方便本地快速啟動與驗證，但不符合生產架構。

**目標**

生產環境需將 business data 獨立到獨立的 `nexus_data` DB，authz policy store 維持在 `nexus_authz`：

```
nexus_authz  ← authz_* 表、PG functions、RLS policy
nexus_data   ← lot_status、sales_order 及其他業務資料表
```

**影響範圍**

- `deploy/docker-compose/docker-compose.yml`：新增第二個 DB 或第二個 service
- `deploy/docker-compose/init-db.sh`：拆分 migration 到不同 DB
- `database/migrations/V014__sample_lot_status.sql`：移至 `nexus_data` 的 init 流程
- `database/migrations/V019__path_c_native_rls.sql`：RLS policy 需在 `nexus_data` DB 內建立
- `deploy/docker-compose/pgbouncer/pgbouncer.ini`：`dbname` 由 `nexus_authz` 改為 `nexus_data`
- `services/authz-api/src/db.ts`：可能需要雙 pool（authz pool + data pool），或確認 `authz_filter()` 回傳的 WHERE clause 由應用層注入到 business DB 查詢
- `authz_filter()` 跨 DB 查詢方式：若 policy store 和 data 分開，business DB 的 RLS 需透過 `dblink` 呼叫 authz function，或改為應用層注入 WHERE clause（兩種方案各有優劣，需決策後記錄 ADR）

**完成標準**

拆分後需重新驗證三條路徑行為：
- Path A：`authz_resolve()` 回傳的 L1 filter 能正確作用在 `nexus_data` 的資料

**備註**：此項目完成後，`docs/nexus-startup-guide.md` 的 docker-compose 範例也需同步更新。

## 二、安全性

### SEC-01　authz_resolve() 回傳 rls_expression 違反 SEC-1

**狀態**：已完成　｜　**優先級**：P1
**完成於**：2026-04-13。resolve.ts 加 sanitizeForClient()，單一 endpoint 依角色控制輸出深度。前端 type 更新。

**現狀**

`authz_resolve()` 的 SQL 實作（V008）直接把 `rls_expression` 包含在 `L1_data_scope` 的 JSON 輸出中，REST API `/api/resolve` 也原封不動地把這份 JSON 回傳給前端。但 `docs/standards/` (old CLAUDE.md removed) 的安全紅線第 2 條明確規定：「Resolved config 不傳送 rls_expression 和 mask function 到客戶端。」目前實作直接違反此規則。

`rls_expression` 暴露給客戶端等於告訴攻擊者資料庫欄位名稱和篩選邏輯，可輔助 SQL injection 攻擊。

**目標**

在 `services/authz-api/src/routes/resolve.ts` 的回傳前，過濾掉 `L1_data_scope` 中每個 policy 的 `rls_expression` 和 `L2_column_masks` 中的 `function`、`pg_function`、`template` 等伺服器端欄位。客戶端只收到：`resource`、`action`、`allowed: boolean`、`mask_type`。

**影響範圍**：`services/authz-api/src/routes/resolve.ts`、前端 `AuthzContext.tsx` 的 `ResolvedConfig` 型別定義

**備註**：這是 roadmap Phase 26（SEC-1）的對應實作。

### SEC-02　deny 規則是否走 cache 的決策矛盾

**狀態**：已完成　｜　**優先級**：P1
**完成於**：2026-04-13。V024 修正 authz_check_from_cache() 加入 deny-wins 語義。authz_resolve() L0_functional 包含 allow + deny。security.md 更新 ADR。

**現狀**

`docs/standards/security.md` 第 6 條規定：「確認 explicit deny 規則不從 cache 讀取，而是即時查詢 DB。」但主架構文件 §VII Mega-Prompt 說明 `authz_check_from_cache()` 評估的是完整 resolved config JSON（含 deny 規則），意味著 deny 也走 cache。兩份文件相互矛盾，且 `authz_check_from_cache()` 尚未實作，此矛盾需在實作前解決。

**目標**

做出明確決策並在兩份文件中統一：
- 方案 A（安全優先）：`authz_check_from_cache()` 只處理 allow 查詢；deny 查詢繞過 cache 即時查 DB
- 方案 B（效能優先）：allow 和 deny 都走 cache，但 deny 的 cache TTL 設為 ≤ 5 秒（需記錄殘留風險）

**影響範圍**：`docs/standards/security.md`、`docs/phison-data-nexus-architecture-v2.4.md` §VII、未來的 `database/migrations/V0xx__cache_functions.sql`

### FEAT-02　資源繼承方向與 authz_check() 實作不符

**狀態**：可關閉（child→parent 方向正確，實測 PE read module:mrp.lot_tracking 可繼承到 table:lot_status）　｜　**優先級**：P3

**現狀**

`authz_check()` 的遞迴 CTE `res_tree` 只往上走（child → parent），代表「對子資源授權隱含對父資源也有授權」。但 seed data（§9.3）和架構文件的意圖是「對父資源（如 `module:authz_admin`）授權，應自動繼承到所有子資源（如各 `page:authz_admin.*`）」，而這個「父 → 子」方向在目前實作中完全不生效。

實際影響：`('AUTHZ_ADMIN', 'read', 'module:authz_admin', 'allow')` 這筆權限，目前對 `page:authz_admin.subjects` 的 `authz_check()` 結果是 FALSE，不是預期的 TRUE。

**目標**

二擇一：
1. 修改 `authz_check()` 加入雙向查詢（parent 的 allow 也能繼承到 children）
2. 在架構文件明確說明「繼承是單向的（child → parent）」，並在 seed data 中把所有父資源授權展開成明確的子資源授權

**影響範圍**：`database/migrations/V007__core_functions.sql`（或新增 migration 覆蓋）、`database/migrations/V013__seed_data.sql`

### FEAT-03　Reversible Deployment / Teardown Sync

**狀態**：待處理　｜　**優先級**：P2
**發現日期**：2026-04-21（在 ds:pg_k8 lifecycle 教學時發現）

**現狀**

Deployment 的四個 step (Sync Local Grants / Sync to Remote DB / Apply PgBouncer Config) 都是 **convergence-only**：只處理 `is_active = TRUE` 的 profile / credential，**不會清除已 deactivate 的 profile 在外部 DB 留下的 role 和 GRANT**。

具體在 `services/authz-api/src/lib/remote-sync.ts:72-76`：

```sql
WHERE dp.is_active AND ds.is_active
```

下次 sync 會跳過 inactive 的 profile，但孤兒 PG role + GRANT 永久留在遠端 Greenplum / 本機 nexus_authz，造成：

1. **權限漂移**：`v_pool_ssot_check` 看不到的孤兒 role
2. **稽核盲點**：DBA 在遠端看到 `nexus_xxx_old_team_ro` 不知道是誰建的、能不能砍
3. **rollback 困難**：使用者誤建一個 profile 想撤回，要手動連遠端 DROP ROLE
4. **管理 UX 差**：deactivate ≠ uninstall，沒有對應的「拆除」操作

**目標**

新增一個 Step 5 "Teardown / Cleanup" 或在現有 Sync 加 `--with-teardown` 旗標：

```sql
-- 加進 syncExternalGrants() 的邏輯
FOR each (role 在遠端存在 AND 對應 profile 已 inactive 或不存在):
    REVOKE ALL ON SCHEMA ... FROM role
    REVOKE ALL ON ALL TABLES ... FROM role
    DROP ROLE role
    log SyncAction { action: 'TEARDOWN_ROLE', ... }
```

UI 上應該：
- Profile 列 → Deactivate 按鈕旁加 **Deactivate & Teardown** 二段式選項
- Deployment 加一個 step「Cleanup Orphans」，列出本機+遠端有但 SSOT 沒有的 role，使用者勾選後執行
- 執行前必須 DangerConfirmModal 列出每一筆會被 DROP 的 role

**安全考量**

`DROP ROLE` 在 Greenplum 若該 role 還有 owned objects（views、tables）會 fail。需要先：
- `REASSIGN OWNED BY <role> TO gpadmin`
- `DROP OWNED BY <role>`
- `DROP ROLE`

**影響範圍**：

- `services/authz-api/src/lib/remote-sync.ts`（新增 teardown logic）
- `services/authz-api/src/routes/pool.ts`（新 endpoint `/sync/teardown`）
- `apps/authz-dashboard/src/components/pool/DeploymentPhase.tsx`（新 step UI）
- `database/migrations/V0xx__teardown_audit_log.sql`（如需獨立 audit）

**業務影響**：高 — 解決了 deployment 「只能往前不能回頭」的根本問題，是 production-ready 的必要功能

### SEC-03　config-exec.ts SQL injection 風險（filter.field + orderBy）

**狀態**：已完成　｜　**優先級**：P1
**完成於**：2026-04-13。filter.field 驗證 via validColumns（SSOT from information_schema），orderBy 用 sanitizeOrderBy() 驗證。

**現狀**

`services/authz-api/src/routes/config-exec.ts` 的 `resolveFilterOptions()` 直接將 `filter.field` 插入 SQL（`SELECT DISTINCT ${filter.field}::text`），`filter.field` 來自 `authz_ui_page.filters_config` 的 JSONB。雖然此資料目前只有管理員可寫入，但未做 column name 驗證。

2. `buildMaskedSelect()` 的 `orderBy` 參數驗證各欄位名存在於查詢表的 column 列表
3. 或統一使用 identifier quoting（`format('%I', col)`）

#### SEC-06a　crypto.ts dev fallback key 加 production 擋板

在 `crypto.ts` `getKey()` 中，當 `NODE_ENV === 'production'` 且 `ENCRYPTION_KEY` 未設定時，直接 throw Error 而非使用 fallback key。防止 production 意外用了可預測的加密金鑰。

**檔案**：`services/authz-api/src/lib/crypto.ts:13-18`

```typescript
function getKey(): Buffer {
  const keyHex = process.env.ENCRYPTION_KEY;
  if (!keyHex || keyHex.length !== 64) {
    if (process.env.NODE_ENV === 'production') {
      throw new Error('ENCRYPTION_KEY env var is required in production (64-char hex)');
    }
    // Dev fallback — deterministic key for POC (NOT for production)
    return Buffer.from('0123456789abcdef...', 'hex');
  }
  return Buffer.from(keyHex, 'hex');
}
```

## 八、開發環境 (DX)

### DX-01　Shell script CRLF 換行符導致 Docker 容器啟動失敗

**狀態**：已完成　｜　**優先級**：P1
**完成於**：2026-04-13。新增 `.gitattributes` 強制 `*.sh` 使用 LF，並 re-normalize 現有 3 個 .sh 檔案。

**現狀**

Windows 環境下 git clone 的 `.sh` 檔案會帶有 CRLF（`\r\n`）換行符。Docker 容器（Linux）執行這些腳本時會報錯：`/bin/bash^M: bad interpreter: No such file or directory`，導致 PostgreSQL 初始化腳本 `init-db.sh` 執行失敗，容器直接 exit(126)。

受影響檔案：
- `deploy/docker-compose/init-db.sh`
- `scripts/verify-milestone1.sh`
- `scripts/verify-path-c.sh`

**目標**

根本解決方案：在 repo 根目錄新增 `.gitattributes`，強制 `.sh` 檔案使用 LF：

```
*.sh text eol=lf
```

臨時解決方案：手動執行 `sed -i 's/\r$//' <file>` 轉換。

**影響範圍**：所有 `.sh` 檔案、新增 `.gitattributes`
