## 一、架構 / 基礎設施

### ARCH-01　Business Database 獨立分離

**狀態**：待處理　｜　**優先級**：P0

**現狀**

目前 business data（`lot_status`、`sales_order`）和 authz policy store（所有 `authz_*` 表）放在同一個 PostgreSQL DB（`nexus_authz`）。這是 POC 階段的設計決策，方便本地快速啟動與驗證，但不符合生產架構。

**目標**

生產環境需將 business data 獨立到獨立的 `nexus_data` DB，authz policy store 維持在 `nexus_authz`：

```
nexus_authz  ← authz_* 表、PG functions、RLS policy
nexus_data   ← lot_status、sales_order 及其他業務資料表
```

**影響範圍**

- `deploy/docker-compose/docker-compose.yml`：新增第二個 DB 或第二個 service
- `deploy/docker-compose/init-db.sh`：拆分 migration 到不同 DB
- `database/migrations/V014__sample_lot_status.sql`：移至 `nexus_data` 的 init 流程
- `database/migrations/V019__path_c_native_rls.sql`：RLS policy 需在 `nexus_data` DB 內建立
- `deploy/docker-compose/pgbouncer/pgbouncer.ini`：`dbname` 由 `nexus_authz` 改為 `nexus_data`
- `services/authz-api/src/db.ts`：可能需要雙 pool（authz pool + data pool），或確認 `authz_filter()` 回傳的 WHERE clause 由應用層注入到 business DB 查詢
- `authz_filter()` 跨 DB 查詢方式：若 policy store 和 data 分開，business DB 的 RLS 需透過 `dblink` 呼叫 authz function，或改為應用層注入 WHERE clause（兩種方案各有優劣，需決策後記錄 ADR）

**完成標準**

拆分後需重新驗證三條路徑行為：
- Path A：`authz_resolve()` 回傳的 L1 filter 能正確作用在 `nexus_data` 的資料
- Path B：`authz_filter()` WHERE clause 能注入到跨 DB 的 API 查詢
- Path C：`nexus_pe_ro` 等 pool role 連線到 `nexus_data` 後 RLS 仍正確觸發

**備註**：此項目完成後，`docs/nexus-startup-guide.md` 的 docker-compose 範例也需同步更新。

### ARCH-02　authz_sync_column_views() 缺失實作

**狀態**：待處理　｜　**優先級**：P1

**現狀**

`V010__path_c_sync.sql`（原版）及主架構文件 §3.6 的 `authz_sync_db_grants()` 中，`denied_columns` 處理部分有一段 `NULL; -- placeholder: column masking views are generated by authz_sync_column_views()`，指向一個從未被定義的函式。V015 已改用 column-level REVOKE 作為替代方案，但 `authz_sync_column_views()` 函式本身以及架構文件中對它的引用從未清理。

**目標**

二擇一：
1. 實作 `authz_sync_column_views()`，為每個有 `denied_columns` 的 pool profile 自動產生 masking view（`CREATE OR REPLACE VIEW v_{table}_{role} AS SELECT ...`）
2. 確認 V015 的 column-level REVOKE 方案已足夠，刪除 placeholder 及架構文件中的相關引用，更新 §3.6 說明

**影響範圍**：`database/migrations/`（新增 migration 或更新 V015）、`docs/phison-data-nexus-architecture-v2.4.md` §3.6

### ARCH-03　Path A/C 共用 RLS 的實作方式未決策

**狀態**：待處理　｜　**優先級**：P2

**現狀**

架構文件 §3.7 說明 Path C 的 sync artifacts 包含「RLS policies on tables (shared with Path A)」。但 `reference_md/authz-sql.md` 規定：Path A 的 RLS USING 子句可用 `current_setting()`，Path C 必須用 `pg_has_role()` 以防 session variable 偽造。同一張表只能有一份 RLS policy，兩個路徑若共用同一份 policy 卻需要不同實作語意，目前沒有解決方案。

V019 的實際做法是用 `current_setting('app.product_line', TRUE)`（Path A 慣用方式），這樣的 Path C RLS 違反了 authz-sql.md 的安全規範（SEC-3）。

**目標**

做出明確決策並記錄 ADR：
- 方案 A：針對同一張表建立兩份 RLS policy（一份給 Path A 角色用 `current_setting`，一份給 Path C pool role 用 `pg_has_role`）
- 方案 B：統一用 `pg_has_role()` 檢查，Path A 也改為用 role-based 判斷而非 session variable

**影響範圍**：`database/migrations/V019`、`docs/phison-data-nexus-architecture-v2.4.md` §3.7、`docs/standards/sql.md`

## 二、安全性

### SEC-01　authz_resolve() 回傳 rls_expression 違反 SEC-1

**狀態**：待處理　｜　**優先級**：P1

**現狀**

`authz_resolve()` 的 SQL 實作（V008）直接把 `rls_expression` 包含在 `L1_data_scope` 的 JSON 輸出中，REST API `/api/resolve` 也原封不動地把這份 JSON 回傳給前端。但 `docs/standards/` (old CLAUDE.md removed) 的安全紅線第 2 條明確規定：「Resolved config 不傳送 rls_expression 和 mask function 到客戶端。」目前實作直接違反此規則。

`rls_expression` 暴露給客戶端等於告訴攻擊者資料庫欄位名稱和篩選邏輯，可輔助 SQL injection 攻擊。

**目標**

在 `services/authz-api/src/routes/resolve.ts` 的回傳前，過濾掉 `L1_data_scope` 中每個 policy 的 `rls_expression` 和 `L2_column_masks` 中的 `function`、`pg_function`、`template` 等伺服器端欄位。客戶端只收到：`resource`、`action`、`allowed: boolean`、`mask_type`。

**影響範圍**：`services/authz-api/src/routes/resolve.ts`、前端 `AuthzContext.tsx` 的 `ResolvedConfig` 型別定義

**備註**：這是 roadmap Phase 26（SEC-1）的對應實作。

### SEC-02　deny 規則是否走 cache 的決策矛盾

**狀態**：待處理　｜　**優先級**：P1

**現狀**

`docs/standards/security.md` 第 6 條規定：「確認 explicit deny 規則不從 cache 讀取，而是即時查詢 DB。」但主架構文件 §VII Mega-Prompt 說明 `authz_check_from_cache()` 評估的是完整 resolved config JSON（含 deny 規則），意味著 deny 也走 cache。兩份文件相互矛盾，且 `authz_check_from_cache()` 尚未實作，此矛盾需在實作前解決。

**目標**

做出明確決策並在兩份文件中統一：
- 方案 A（安全優先）：`authz_check_from_cache()` 只處理 allow 查詢；deny 查詢繞過 cache 即時查 DB
- 方案 B（效能優先）：allow 和 deny 都走 cache，但 deny 的 cache TTL 設為 ≤ 5 秒（需記錄殘留風險）

**影響範圍**：`docs/standards/security.md`、`docs/phison-data-nexus-architecture-v2.4.md` §VII、未來的 `database/migrations/V0xx__cache_functions.sql`

## 三、功能缺失

### FEAT-01　authz_check_from_cache() 定義缺失

**狀態**：待處理　｜　**優先級**：P2

**現狀**

`CLAUDE.md` 的 migration 表格（V007）和架構文件 §VII Mega-Prompt 均提到 `authz_check_from_cache()` 函式，但整份 migration 中找不到此函式的任何 SQL 定義。目前 V007 實際只包含 `_authz_resolve_roles`、`authz_check`、`authz_filter` 三個函式。

**目標**

實作 `authz_check_from_cache(resolved_config JSONB, action TEXT, resource TEXT) RETURNS BOOLEAN`，邏輯為：直接從已 resolve 的 JSON 中查找 L0_functional，不發出任何 DB 查詢（標記為 `IMMUTABLE`）。此函式是 Phase 22 兩層 cache 架構的核心依賴。

**影響範圍**：新增 `database/migrations/V0xx__authz_check_from_cache.sql`，需先解決 TECH-DEBT SEC-02（決定 deny 是否走 cache）

### FEAT-02　資源繼承方向與 authz_check() 實作不符

**狀態**：待處理　｜　**優先級**：P2

**現狀**

`authz_check()` 的遞迴 CTE `res_tree` 只往上走（child → parent），代表「對子資源授權隱含對父資源也有授權」。但 seed data（§9.3）和架構文件的意圖是「對父資源（如 `module:authz_admin`）授權，應自動繼承到所有子資源（如各 `page:authz_admin.*`）」，而這個「父 → 子」方向在目前實作中完全不生效。

實際影響：`('AUTHZ_ADMIN', 'read', 'module:authz_admin', 'allow')` 這筆權限，目前對 `page:authz_admin.subjects` 的 `authz_check()` 結果是 FALSE，不是預期的 TRUE。

**目標**

二擇一：
1. 修改 `authz_check()` 加入雙向查詢（parent 的 allow 也能繼承到 children）
2. 在架構文件明確說明「繼承是單向的（child → parent）」，並在 seed data 中把所有父資源授權展開成明確的子資源授權

**影響範圍**：`database/migrations/V007__core_functions.sql`（或新增 migration 覆蓋）、`database/migrations/V013__seed_data.sql`
